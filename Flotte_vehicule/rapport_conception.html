2. Justification des choix de conception

Dans cette partie, on explique pourquoi on a choisi d'utiliser une classe abstraite à certains endroits, et une interface à d'autres.

Pourquoi Mission est une classe abstraite ?

Dans notre application, une mission est toujours soit courte, soit longue. Il n'existe pas de mission "générique" sans type précis. On a donc rendu la classe Mission abstraite pour qu'il soit impossible de créer un objet Mission directement.

De plus, la méthode genererRapport() est déclarée abstraite dans Mission car le rapport d'une mission courte n'est pas le même que celui d'une mission longue. Cela oblige chaque sous-classe à écrire sa propre version du rapport.

Pourquoi Assignable est une interface ?

Assignable représente un comportement : "est-ce que cette entité est disponible ?". Ce comportement peut concerner aussi bien un chauffeur qu'un véhicule, alors qu'ils n'ont pas de lien entre eux.

Une interface permet d'ajouter ce comportement à n'importe quelle classe, sans les forcer à avoir la même classe parente. C'est plus flexible qu'une classe abstraite dans ce cas.

Pourquoi Trackable est une interface ?

Le suivi de position (savoir où se trouve le chauffeur en cours de route) n'est utile que pour les missions longues. Une mission courte n'a pas besoin de ça.

En créant l'interface Trackable, on peut ajouter ce comportement uniquement à MissionLongue, sans toucher à la classe abstraite Mission ni à MissionCourte.

Pourquoi un générique borné pour GestionnaireOperationnel ?

On a écrit : GestionnaireOperationnel<T extends Mission>. Cela signifie que le gestionnaire peut travailler avec n'importe quel type de mission (courte ou longue), du moment que c'est bien une Mission.

Grâce à ça, on n'a pas besoin d'écrire deux gestionnaires différents. Un seul suffit pour gérer les deux types, tout en restant fiable au niveau du type.

Pourquoi Incident est une classe abstraite ?

Comme pour Mission, un incident ne peut pas exister sans type précis. Dans notre application, un incident est toujours soit une panne mécanique, soit un accident. Rendre Incident abstraite empêche de créer un incident "générique" sans information concrète.

La méthode genererRapport() est déclarée abstraite car le rapport d'une panne mécanique (qui mentionne le type de panne) n'est pas le même que celui d'un accident (qui mentionne la gravité). Chaque sous-classe est donc obligée d'écrire sa propre version.

Pourquoi Maintenable est une interface ?

Maintenable représente un comportement : "cet objet peut subir une maintenance, signaler des incidents et fournir son historique". Ce comportement est prévu pour être implémenté par la classe Vehicule, développée par une autre partie du groupe.

Utiliser une interface plutôt qu'une classe abstraite permet au futur Vehicule d'hériter de sa propre hiérarchie (véhicule léger, lourd, spécial) tout en respectant le contrat de maintenance. C'est la troisième interface métier requise par le cahier des charges, avec Assignable et Trackable.

3. Difficultés rencontrées et solutions adoptées

Difficulté 1 — Conflit de versions (jakarta vs javax)

Problème : Le projet utilisait une version récente de l'API Java EE (jakarta.servlet), mais le plugin Tomcat qu'on avait configuré dans Maven ne la reconnaissait pas. L'application refusait de démarrer.

Solution : On a changé la dépendance dans le fichier pom.xml pour utiliser l'ancienne version (javax.servlet), compatible avec notre version de Tomcat. On a aussi mis à jour tous les imports dans les Servlets.

Difficulté 2 — Erreur dans la JSP à cause d'un mauvais nom de variable

Problème : Dans la page JSP, on avait nommé la variable de boucle c, comme le préfixe JSTL (prefix="c"). Cela créait un conflit et les données ne s'affichaient pas.

Solution : On a renommé la variable de boucle en chauffeur, ce qui a supprimé l'ambiguïté et fait fonctionner l'affichage correctement.

Difficulté 3 — Partager les données entre plusieurs Servlets

Problème : On avait trois Servlets différents (missions, chauffeurs, statistiques) qui devaient tous accéder aux mêmes données. Sans base de données, comment faire en sorte qu'ils partagent la même liste de missions et de chauffeurs ?

Solution : On a utilisé le pattern Singleton : le GestionnaireOperationnel ne peut avoir qu'une seule instance dans toute l'application. Tous les Servlets y accèdent via GestionnaireOperationnel.getInstance().

Difficulté 4 — Partager les incidents entre deux Servlets

Problème : Le IncidentServlet et le StatistiquesServlet devaient tous les deux accéder à la même liste d'incidents. Sans base de données, le risque était que chaque Servlet crée sa propre instance de GestionnaireMaintenance avec des données différentes.

Solution : On a appliqué le même pattern Singleton que pour GestionnaireOperationnel : la méthode GestionnaireMaintenance.getInstance() garantit qu'une seule instance existe dans toute l'application. On a aussi ajouté une méthode setInstance() pour permettre de restaurer les données sauvegardées depuis le fichier de persistance au démarrage.

Difficulté 5 — Le port 8080 déjà utilisé

Problème : En relançant Tomcat plusieurs fois, on obtenait l'erreur "Address already in use: bind :8080" car un processus précédent occupait encore le port.

Solution : Attendre que le processus précédent se ferme complètement, ou vérifier dans le gestionnaire des tâches qu'aucun Java ne tourne encore en arrière-plan.

4. Ce qui aurait été amélioré avec plus de temps

Amélioration Pourquoi c'est utile
Ajouter une base de données (MySQL) Pour l'instant les données sont perdues si on modifie le code. Une vraie base de données garderait les données de manière permanente et fiable.
Ajouter une page de connexion Dans une vraie application, tout le monde ne devrait pas pouvoir supprimer ou modifier les données. Un système de login avec des rôles (admin / utilisateur) serait nécessaire.
Mieux vérifier les formulaires On vérifie que les champs obligatoires sont remplis, mais on pourrait aller plus loin : empêcher un itinéraire vide, limiter la longueur des textes, etc.
Écrire des tests (JUnit) Les méthodes de filtrage et de tri fonctionnent, mais on n'a pas eu le temps d'écrire des tests automatisés pour les vérifier à chaque modification du code.
Exporter les données en CSV Permettre à l'utilisateur de télécharger la liste des missions ou des chauffeurs dans un fichier Excel serait très pratique.
Relier les modules entre eux Aujourd'hui les missions, les véhicules et les incidents sont gérés séparément. Idéalement, une mission serait liée à un véhicule précis, et un incident pourrait être rattaché à une mission.

En résumé, le projet répond aux exigences du cahier des charges : classes abstraites, interfaces, générique borné, collections avancées, streams et lambdas, CRUD complet, filtrage, statistiques, persistance et interface web. Les améliorations listées ci-dessus nécessiteraient davantage de temps et des connaissances plus avancées.
